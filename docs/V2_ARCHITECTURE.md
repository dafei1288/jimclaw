# JimClaw V2 架构反思与重构设计：从黑盒巨石到微服务化

**日期**: 2026-03-10
**状态**: 反思与规划阶段

## 1. 痛点反思：我们陷入了什么误区？

在目前的架构中，JimClaw 被设计成了一辆“一旦启动就无法停车的高速列车”。

- **极度耦合的“黑盒”**: 从输入一句话需求，到需求拆解、架构设计、代码生成、测试、部署，全部绑定在一个长达数百行的 `StateGraph` 流水线中。
- **调试成本极高**: 如果最后一步部署失败，我们往往需要从头再跑一遍需求分析和代码生成。这种“全有或全无”的执行模式不仅浪费大量 Token，还让排查单点问题变得极其困难。
- **缺乏阶段性干预**: 虽然我们有 `approval` 节点，但整体上人类很难在中间环节（比如 PM 刚写完契约，或者架构师刚画完图）把数据拿出来修改，然后再单独喂给下一道工序。
- **系统过于庞大**: `createJimClawGraph` 变成了一个上帝对象，管理着所有状态。

**结论**：目前的 JimClaw 是一个**巨石架构 (Monolithic)** 的黑盒应用，它违背了软件工程中“高内聚、低耦合”的原则，不利于分阶段测试和稳定迭代。

## 2. V2 架构演进方向：RESTful 服务化拆分

正如你所建议的，我们必须将这头大象拆解。将各个核心环节独立成一个个“纯函数”式的 REST 服务。每个服务只负责接收明确的上下文，输出特定的 Artifact，互不干扰。

### 2.1 核心思想：状态外置与步骤解耦
- **废弃巨石 Graph**：不再使用一个从头跑到尾的 LangGraph 实例控制一切。
- **状态持久化驱动**：前端或外部调用方掌握进度，通过调用不同的 API 端点来驱动项目前进。上一步的输出就是下一步的输入。

### 2.2 拟定的 REST API 矩阵

#### 服务一：需求解析服务 (PM Service)
- **Endpoint**: `POST /api/v2/design/contract`
- **Input**: `{ "goal": "做一个图书管理系统" }`
- **Output**: 返回结构化的 `TaskContract` (JSON)。
- **优势**: 开发者可以在拿到契约后，手动调整验收标准，直到满意为止，再进入下一步。

#### 服务二：架构设计服务 (Architect Service)
- **Endpoint**: `POST /api/v2/design/spec`
- **Input**: `{ "contract": {...} }`
- **Output**: 返回 `TechSpec`, `SystemManifest` 和 `ApiContract`。
- **动作**: 架构师在此处独立完成端口探测等前置准备。

#### 服务三：代码生成服务 (Coder Service)
- **Endpoint**: `POST /api/v2/build/code`
- **Input**: `{ "spec": {...}, "tasks": [...] }` (可分批次传入单任务或多任务)
- **Output**: 写入磁盘，返回生成的代码快照和执行日志。

#### 服务四：质量审计服务 (QA Service)
- **Endpoint**: `POST /api/v2/verify/qa`
- **Input**: 触发对指定 `workspace` 的单元测试。
- **Output**: 返回详细的 `IssueTracker` 工单列表。

#### 服务五：部署服务 (Deploy Service)
- **Endpoint**: `POST /api/v2/run/deploy`
- **Input**: 指定 `workspace` 和启动参数。
- **Output**: 启动容器并返回健康检查后的访问 URL。

## 3. 实施路径 (Action Plan)

1. **改造底座**：剥离 `src/core/nodes/` 下的函数，使其不再强依赖 LangGraph 的上下文回调，而是变成标准的 `(input) => output` 的异步函数。
2. **重写 Server**：在 `src/server.ts` 中引入 Express Router，实现上述 5 个核心 API。
3. **前端/CLI 重构**：修改调用方式。不再是触发一个 `run-task` 等到底，而是分步骤调用。支持单步测试和单步重试。
4. **保留串联模式**：为了体验，可以写一个外层 Orchestrator 脚本，依次调用这 5 个 API，但底层已经彻底解耦。

## 4. 预期收益
- **精准隔离**：我可以单独写一个脚本，疯狂测试“代码生成服务”，而不必每次都跑环境搭建。
- **可复用性**：这些独立的 Agent 接口可以被接入其他平台，比如直接给 IDE 插件提供“生成契约”的能力。
- **不再掩耳盗铃**：每一步的结果都必须经过严格的 Schema 校验才能流向下一个 API。
